# Backend

데이터베이스 프로그래밍 프로젝트 백엔드 저장소입니다.

## 프로젝트 규칙

### 1. 커밋 컨벤션

커밋 메시지는 `태그: 한글 커밋 메시지` 형식으로 작성합니다.

예시:

```text
fix: 로그인 API에서 NullPointerException 발생 문제 해결
```

| 태그 | 설명 |
| --- | --- |
| feat | 새로운 기능 구현 |
| add | feat 이외의 부수적인 코드 추가 및 라이브러리 추가, 새로운 파일 생성 |
| mod | feat 이외의 코드 수정 및 내부 파일 수정 |
| rename | 파일 혹은 폴더명 수정 |
| del | 사용하지 않거나 쓸모없는 코드나 파일, 폴더 삭제 |
| fix | 버그 및 오류 해결 |
| !breaking change | 커다란 API 변경 |
| !hotfix | 급하게 치명적인 버그를 고친 경우 |
| style | 코드 포맷팅, 세미콜론 누락, 오타 수정, 탭 사이즈 변경 |
| refactor | 프로덕션 코드 리팩토링, 기능 변경 없이 현재 구현 개선, 변수명 변경 |
| test | 테스트 추가, 테스트 리팩토링 |
| design | CSS 등 사용자 UI 디자인 변경 |
| docs | README나 Wiki 등의 문서 개정 |
| chore | 빌드 업무 수정, 패키지 매니저 수정, dotenv 요소 변경 등 |

### 2. PR 규칙

PR 제목은 `[Type] PR 제목` 형식으로 작성합니다.

예시:

```text
[Feat] 회원가입 및 로그인 기능 개발
```

- 타입은 대문자로 작성합니다.
- label을 설정합니다.
- 작업자 자신을 Assign합니다.
- Code Review를 요청합니다.
- 작성자 외 1명이 확인하면 작성자가 merge합니다.

PR 본문은 다음 항목을 포함합니다.

```md
# 구현 기능
- 구현한 기능을 요약하여 정리합니다.

# 구현 상태 (선택)
- img, gif, video 또는 내용 정리

# Resolve
- 이슈 태그(ex: #7)
```

### 3. Issue 규칙

- 각 기능에 맞는 Issue 템플릿을 사용합니다.
- 작업 및 변경사항 확인용으로 Issue를 작성합니다.
- to-do에 구현해야 할 기능을 작성합니다.
- 구현이 끝난 항목은 체크 표시합니다.

### 4. Branch 규칙

브랜치 네이밍 규칙은 `feat/(기능 이름)/(이슈 번호)` 형식입니다.

예시:

```text
feat/user-login/11
```

- 같은 기능이더라도 수정사항이 생기면 새로운 브랜치를 생성합니다.
- merge된 브랜치는 삭제합니다.
- merge 흐름은 `feat -> develop -> main` 순서로 진행합니다.
- `feat`: 각 기능을 개발하는 브랜치입니다.
- `develop`: 각 기능 테스트 완료 후 병합하는 브랜치입니다.
- `main`: 배포되는 브랜치입니다.
